Design Viewer vs Sim Viewer — Architecture

How should the viewer conceptually split?

Current Architecture

Today: Everything runs through MuJoCo
viewer.ts (router)
?bot=X → bot-viewer.ts (MuJoCo)
?component=X → component-browser.ts (Three.js)
Bot Viewer (MuJoCo WASM)
• Loads bot.xml + meshes through MuJoCo
• Per-body visibility only ⛔
• Single color per geom ⛔
• Multi-material = hack overlay ⛔
• Modes: Explore, Joint, Assembly, IK
• Component tree can't control individual parts ⛔
Component Browser (Three.js)
• Loads STLs directly — no MuJoCo
• Per-layer visibility ✅
• Design layers (bracket, coupler, clearance) ✅
• Standalone components only ⛔
• No assembled bot view ⛔

Proposed Architecture

New: Design Viewer (Three.js) + Sim Viewer (MuJoCo)
viewer.ts (router)
?bot=X → Design tab (default) + Sim tab
?component=X → Design tab (component scope)
Design Viewer (Three.js — no MuJoCo)
• Loads manifest + STLs directly
• Per-mesh visibility ✅
• Multi-material native ✅
• Design layers (bracket, coupler, clearance) ✅
• Component tree with ● / ○ toggles ✅
• Wiring + fastener overlays ✅
• Works at any scope: component → assembly → bot ✅
SCOPE LEVELS:
Full bot → Assembly → Body → Component (zoom in/out)
Sim Viewer (MuJoCo WASM)
• Joint control + IK
• Physics simulation
• Controller testing
• Collision geometry
• Body-level visibility (fine here)
KEEPS:
Joint mode, IK mode, Assembly mode
Shared Infrastructure
• SceneTree (node visibility)
• Component tree panel
• Section cutter
• Camera / controls
✓

This split looks right

Design viewer + Sim viewer as separate tabs, shared component tree and camera
